## 2026-07-16 v6.1.1 — 07 why_read 长度弹性化（v6.1.0 生产抽验修补，用户定方向）

- **背景**：v6.1.0 部署后 ECS 抽验 5 簇：内容质量全达标（数字密集/零评价词/四层结构），但 3/5 超 240 字护栏被硬截断尾句截半。用户拍板：**不设死上限，长度随材料伸缩**——薄材料 100 字就收，厚材料要 200 字就给 200 字；卡片超 3 行由前端省略号兜底（line-clamp 已上线）。
- **改动**（v6.1.0→v6.1.1，仅 why_read 长度语义）：
  1. "120-180 字"全部替换为"常规 100-200 字、按材料信息量伸缩"；保留 60 字薄材料下限与禁注水。
  2. 细则加长度段：四层写全即收笔、细节归 summary/breakdown；软上限约 220 字（超出会被系统句边界截断）；无论长短必须完整句收尾。
  3. 自检 #9 改为"长度与材料相称且完整句收尾"。
- **配套（代码侧，已完成待合并）**：`_normalize_why_read` 超 240 字改句边界回退截断（前 240 字内最后一个。！？；处收口），杜绝存量截半句。
- **验证建议**：重回填后统计长度分布（预期集中 100-220）、超 240 触发率（目标 <5%）与完整句收尾率。

## 2026-07-16 v6.1.0 — 07 why_read 改版：teaser → 精简事实摘要

- **背景**：v27 上线当天用户否定 40-80 字"点开理由"形态（信息太薄、与标题重合）。复核 AIHOT 公开 API 50 条实测：其卡片主体是**客观摘要层**（中位 163 字、2-4 句、纯事实数字密集），"推荐理由"是独立评价层——v27 决策 #5/#15 学错了层。用户给出理想样例=AIHOT 摘要层原文。
- **改动**（`07_cluster_summary.md` v6.0.0→v6.1.0，schema 不变仅语义变更）：
  1. why_read 规格：40-80 字三层（增量/所以呢/关注点）→ **120-180 字四层**（主体+核心事件 → 机制/细节 → 量化载荷 → 可选影响/展望）；定位从"点开理由"改为"压缩到 3 行的事实摘要"，在 summary/breakdown 后写。
  2. 禁用清单调整：空泛评价升级为**评价与推荐话术全禁**（评价一个字不写，含"第一次有了 XX"类）；"复读标题"放宽为"禁逐字照抄，首句应为标题的完整句展开"；新增**注水凑字**禁令（材料极薄下限 60 字，允许用原文背景事实补足）。
  3. 新增形式约束：纯文本（不加粗/不分行/不序号，展示端不渲染 Markdown）；"原文有数字则至少一个必须进 why_read"硬门。
  4. Few-shot why_read 重写为 140 字四层示例；自查示范、任务说明、执行步骤 7/自检 9、JSON 模板、字段约束同步。
- **配套（代码侧，另行走 codex-cli）**：`_normalize_why_read` 截断护栏 200→240 字（180 上限只留 20 字余量会截半句）；前端卡片 clamp 移动端 2→3 行；存量重回填需先置空旧 why_read（回填脚本只补 NULL）。
- **验证建议**：temp≤0.3；抽 10 个生产簇（含 1 个单句推文薄材料）双跑：长度落 120-180 区间率、数字进摘要率、评价词正则 0 命中；薄材料簇验证不注水。

## 2026-07-03 v5.1 — 07_cluster_summary.md 长文自适应拆解(路线 A)

**背景**：
用户反馈事件总结"读起来没收获"。对比语鲸对同一篇大会综述的总结(~2000 字、按原文章节分节、每条 bullet 是"人名:断言+证据")发现三个根因：①字数上限 250-450(最多 800)把长文压成目录级信息；②"事件脉络/影响范围"新闻框架套不上大会综述/长教程；③few-shot 是 100 字短事件,锚定短输出。数据:近 14 天可见事件中单来源长文(>3000 字)约占 18%,短内容占 82% 且现状长度合适,故只做长文自适应,不全局加长。

**改动清单：**
- 执行步骤新增第 5 步"判断内容形态":主导来源为长文(深度文章/大会实录/长教程/长访谈,>约 3000 字)时进入长文拆解细则。
- 新增「长文拆解细则」小节：跟随原文自身结构分节(不硬套新闻框架)；观点类内容用断言式 bullet(`**主体：核心断言**——证据`)；覆盖所有主要讲者/章节；附节选示例。
- 字数改为分档：普通事件维持 250-450(丰富时 600-800)；长文可扩展到 1500-2000 字。
- 长文长度控制用结构预算而非纯字数(实测纯字数上限对 MiniMax-M2.7 约束力弱)：最多 7 个小节、每节 2-4 条 bullet、每条 ≤70 字、速览 300-450 字、次要观点并条。
- breakdown 细则新增"禁止目录式罗列"(人数/场次/议题清单最多一笔带过)。
- 配套：`summary_writer.py` 两处 `max_tokens` 4096→8192,防止 2000 字输出 JSON 截断。

**实测(8 个真实样本,4 轮措辞迭代)：**
- 长文按原文结构分节、断言式 bullet 生效;短讯长度不变(515/476 字);议程类目录罗列被砍(1264→1049 字)。
- 长度收敛:7/8 样本进预算;已知限制:极端密度输入(如 45k 字、15 个事件打包的季度综述)仍会超到 ~4100 字,模型不肯压缩单条 bullet,继续磨措辞性价比低,如需强制可后续加代码级压缩。

**预期效果：**
- 长文事件的拆解从"目录信息"变为"一行一个观点",主要讲者/章节全覆盖。
- 短讯/普通事件输出长度与风格不变。
- 输出 token 成本仅在长文类(~18%)上升,整体约 +30-50%。

## 2026-04-27 v15.1 — review/v15.1 收尾（07_cluster_summary.md 关键字使用约束）

**背景**：
forge-review 发现 V2.3 §16.2 warnings 关键字白名单 (`不属于同一事件 / 不构成事件聚合
/ 主体不一致 / 事件不一致 / 无法构成同一事件 / 非跨源事件`) 与 prompt 中"偏题来源
不属于'不构成事件聚合'，应该写 warnings 但不命中关键字"的指引存在弱耦合：LLM 在
描述**单条来源**偏题时，自然语言里很容易出现"src 5 不属于同一事件"或"个别来源主体
不一致"等表述，子串命中会被 `summary_writer._check_invalid_warnings` 判定为

**改动清单：**
- 07_cluster_summary.md `warnings 细则` 末尾补充「关键字使用约束（避免误触发 cluster
  隐藏）」段，明确：6 个白名单关键字**只在 cluster-level 否定结论时使用**，描述单源
  偏题/单源时间过远等 source-level 问题时禁止使用这些字串，改用 `偏题 / 个别来源 /
  src N 内容偏离主题 / 时间窗口外`。
- 解释了系统做子串匹配的实现细节，让 LLM 理解约束的工程动机。

**预期效果：**
- Stage 4 warnings 命中关键字 → cluster 不可见的兜底链路只在真正 cluster-level 错合
  时触发，不再因 LLM 描述单源偏题误伤正常 cluster。
