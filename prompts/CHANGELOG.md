## 2026-07-04 v5.2 — 07/08/02/02_asr 加粗密度·占位语·卡片首句·标题 cap(四界面调研首批)

**背景**：
四界面展示调研(审阅页 A4/A5/B1/B3)实测 40 个事件样本发现四类生成侧问题:①加粗密度失控——02 指令性要求"每个关键实体加粗"是根因,07 对逗号枚举串无约束(20/40 样本出现"加粗占行>55%"的整串枚举加粗,极端样本单句 12 个加粗);②占位语——"GitHub star 相关信息待补充"类待办表述,以及 07 "必须设关键信息小节"+"每小节≥2条"在无资源场景冲突逼出"无具体资源链接"凑数 bullet;③精选卡摘要中段截断——speedReview(median 207 字)按弹窗阅读体写,前端 line-clamp-2/3 只显示 ~40-70 字,省略号落在词中间;④标题超长——7/40 超 30 字 cap(max 45,英文 repo 全名+中文副标题全塞)。

**改动清单：**
- `02_summary_breakdown.md` / `02_summary_breakdown_asr.md`:删除"对关键实体使用加粗标记"与"子要点关键实体同样加粗"两条指令性规则,改为价值驱动措辞(每句 0-1 个重点、枚举串不逐项加粗);新增"未知即省略"占位语禁令;few-shot 示例输出同步降低加粗密度。
- `07_cluster_summary.md`:重点信息选择规范新增"枚举/并列清单不逐项加粗";注意事项新增占位语禁令(保留"明确拒绝披露本身是要点"豁免);"关键信息"小节由硬性"必须设置"改为条件式(无可列资源整节省略,禁占位 bullet);summary 细则新增"首句 ≤40 字自成摘要"(适配列表卡 clamp 截断);title cap 12-30→12-26 字(英文按实际长度计入)+长名精简规则(3 处同步)。
- `08_cluster_title.md`:标题 cap ≤30→≤26 字,同步长名精简规则。

**实测(8 样本×2 轮全量+1 轮定点,评测材料在会话 scratchpad eval/compare_v52.md)：**
- 立即生效:占位语与凑数小节基本清零(仅 61839 纯链接样本正文残留 1 句);枚举串逐项加粗消失(61894 单句 12 处→最多 1 处);长文速览 668→307 字进预算;对照组无回退。
- 3 轮措辞迭代:①纯数字上限(首句≤40/标题 12-26)约束力弱,复现 v5.1 结论,改结构措辞("只写主体+动作""只写一个分句""人名头衔二选一")+few-shot 首句超短句锚定;②第 1 轮出现"来源仅提供标题"类元评论输入质量的新占位变体,加禁令并指向 warnings;③warnings 引导收紧为"完全无正文才用",避免把有内容的边缘样本推去隐藏。
- 已知限制(观察一周):首句实际 48-140 字(软达成,截断体验已显著改善,不行再上前端截句方案);长英文专名标题仍可能超 26 字(max 41);低信息边界样本(61904 型)的"不建议展示"翻转为既有路径的模糊地带,非本轮引入。

## 2026-07-03 v5.1 — 07_cluster_summary.md 长文自适应拆解(路线 A)

**背景**：
用户反馈事件总结"读起来没收获"。对比语鲸对同一篇大会综述的总结(~2000 字、按原文章节分节、每条 bullet 是"人名:断言+证据")发现三个根因：①字数上限 250-450(最多 800)把长文压成目录级信息；②"事件脉络/影响范围"新闻框架套不上大会综述/长教程；③few-shot 是 100 字短事件,锚定短输出。数据:近 14 天可见事件中单来源长文(>3000 字)约占 18%,短内容占 82% 且现状长度合适,故只做长文自适应,不全局加长。

**改动清单：**
- 执行步骤新增第 5 步"判断内容形态":主导来源为长文(深度文章/大会实录/长教程/长访谈,>约 3000 字)时进入长文拆解细则。
- 新增「长文拆解细则」小节：跟随原文自身结构分节(不硬套新闻框架)；观点类内容用断言式 bullet(`**主体：核心断言**——证据`)；覆盖所有主要讲者/章节；附节选示例。
- 字数改为分档：普通事件维持 250-450(丰富时 600-800)；长文可扩展到 1500-2000 字。
- 长文长度控制用结构预算而非纯字数(实测纯字数上限对 MiniMax-M2.7 约束力弱)：最多 7 个小节、每节 2-4 条 bullet、每条 ≤70 字、速览 300-450 字、次要观点并条。
- breakdown 细则新增"禁止目录式罗列"(人数/场次/议题清单最多一笔带过)。
- 配套：`summary_writer.py` 两处 `max_tokens` 4096→8192,防止 2000 字输出 JSON 截断。

**实测(8 个真实样本,4 轮措辞迭代)：**
- 长文按原文结构分节、断言式 bullet 生效;短讯长度不变(515/476 字);议程类目录罗列被砍(1264→1049 字)。
- 长度收敛:7/8 样本进预算;已知限制:极端密度输入(如 45k 字、15 个事件打包的季度综述)仍会超到 ~4100 字,模型不肯压缩单条 bullet,继续磨措辞性价比低,如需强制可后续加代码级压缩。

**预期效果：**
- 长文事件的拆解从"目录信息"变为"一行一个观点",主要讲者/章节全覆盖。
- 短讯/普通事件输出长度与风格不变。
- 输出 token 成本仅在长文类(~18%)上升,整体约 +30-50%。

## 2026-04-27 v15.1 — review/v15.1 收尾（07_cluster_summary.md 关键字使用约束）

**背景**：
forge-review 发现 V2.3 §16.2 warnings 关键字白名单 (`不属于同一事件 / 不构成事件聚合
/ 主体不一致 / 事件不一致 / 无法构成同一事件 / 非跨源事件`) 与 prompt 中"偏题来源
不属于'不构成事件聚合'，应该写 warnings 但不命中关键字"的指引存在弱耦合：LLM 在
描述**单条来源**偏题时，自然语言里很容易出现"src 5 不属于同一事件"或"个别来源主体
不一致"等表述，子串命中会被 `summary_writer._check_invalid_warnings` 判定为
**整个 cluster 不构成事件聚合**，导致正常事件被错误隐藏。

**改动清单：**
- 07_cluster_summary.md `warnings 细则` 末尾补充「关键字使用约束（避免误触发 cluster
  隐藏）」段，明确：6 个白名单关键字**只在 cluster-level 否定结论时使用**，描述单源
  偏题/单源时间过远等 source-level 问题时禁止使用这些字串，改用 `偏题 / 个别来源 /
  src N 内容偏离主题 / 时间窗口外`。
- 解释了系统做子串匹配的实现细节，让 LLM 理解约束的工程动机。

**预期效果：**
- Stage 4 warnings 命中关键字 → cluster 不可见的兜底链路只在真正 cluster-level 错合
  时触发，不再因 LLM 描述单源偏题误伤正常 cluster。

## 2026-04-27 v15.1 — 事件聚合 V2 Stage 4 重写（07_cluster_summary.md）

**背景**：
v15.0 的 cluster summary 是单段平铺 markdown，前端 `ClusterSummaryBlock` 直接
渲染整段。V2.3 §7 / §16.3 决策：summary 必须改为「【精华速览】+【全文拆解】」
双段结构，与单 doc 详情页 `SummaryBlock` 视觉一致；字数收紧到 250-450 字；
warnings 关键字白名单进入 prompt 显式说明，让 Stage 4 兜底降级有可靠输入。

**改动清单：**

### 07_cluster_summary.md（重写）
- 双段结构 `【精华速览】... \n\n【全文拆解】 1. 主题\n- 要点 ...`，硬规则
- 字数 250-450 字（V2.3 §16.3）
- warnings 显式列举 6 个允许关键字（V2.3 §16.2 / feature-spec R5.2）：
  `主体不一致 / 事件不一致 / 无法构成同一事件 / 非跨源事件 / 不属于同一事件 / 不构成事件聚合`
- key_points 4-7 条软规则不变
- system prompt 不含 `{cluster_docs}` 占位符（Eng-A `73dcef7` 已修，本次保留）
- title 细则补充：要让用户在列表中快速判断主体和动作
- 偏题来源 vs 整体非事件区分说明：偏题写 warnings 但不命中关键字

**预期效果：**
- 前端 `ClusterSummaryBlock` 拆双段视觉，复用单 doc `SummaryBlock` 体验
- Stage 4 兜底降级（warnings 命中关键字 → is_visible_in_feed=0）有稳定来源
- 字数收紧让卡片预览不再溢出

## 2026-04-27 v15.1 — 事件聚合 V2 Stage 2 改造

**背景**：
v15.0 的 Stage 2 是单对单 boundary judge（cosine 0.70-0.85 区间调一次 LLM
判 new doc 与 cluster 单条 member 是否同事件）。V2 把 Stage 1 召回放开
为 top-10，Stage 2 改成一次大调用判断 top-10 是否同事件，让 LLM 在候选
之间做相对比较，并由代码层基于 confidence/relationship/cosine 做最终选择。

**改动清单：**

### 10_cluster_top10_judge.md（新增）
- 一次大调用判 New Doc 与 top-10 候选 cluster 是否同事件
- 输入变量：`{new_doc}` + `{candidate_clusters}`
- 输出 JSON `{new_doc_fingerprint, matches[]}`，每个 match 含
  `cluster_id / same_event / confidence / relationship / subject_check /
  action_check / time_check / rationale`
- 显式禁止 LLM 输出 `selected_cluster_id / decision /
  possible_merge_candidates / should_be_visible_if_single_doc`（代码层负责）
- 多源转发同一具体项目（同一 GitHub repo / 产品 / 教程）算 same_event
  （V2 §18.1 Q14 决策）

### 09_cluster_merge_decision.md（DEPRECATED）
- 标 deprecation comment，文件保留作 observation 期参考
- `merge_detector.py` 仍在但 V2 主线不调，新代码不应 load 这个 prompt
- 后续观察期结束后再决定是否彻底删除

## 2026-03-23 v1.1 — 质量提升：规范驱动 + 数据保真

**背景**：
调研 Anthropic/OpenAI 官方 prompt engineering 最佳实践后，对现有 prompt 进行系统性升级。
核心问题：摘要质量不稳定（短内容注水、数据被模糊化、语言不一致）、评分漂移（缺锚点）。

**改动清单：**

### 02_summary_breakdown.md（大改）
- 角色升级：从"信息精选助手"升级为"资深科技编辑"，明确目标（让读者不需要点开原文）
- 新增语言规则：输出一律中文，英文专有名词保留原文
- 新增数据保真规则：数字/百分比/日期/版本号/人名/公司名必须原样保留，禁止模糊替代
- 新增反面约束："不要以'本文介绍了'开头"、"不要加'值得关注'等主观评价"
- 新增内容分流：< 300 字不输出全文拆解，1500+ 字增加话题数
- 新增分类感知：通过 `{category}` 变量，不同分类用不同拆解策略（tutorials 按步骤、tools 按功能）
- 新增 few-shot 示例：一个 industry 类的完整输入/输出样本
- 代码改造：`generate_summaries.py` 从 DB 读取 category 字段注入 prompt

### 01_classify_and_score.md（中改）
- 评分从"四维度输出一个模糊总分"改为"单一综合分 + 5 级锚点"
- 每个分数段有具体标准（9-10=原创深度+一手数据，5-6=准确转述无独立观点，1-2=标题党/纯营销）
- 关键词提取加约束："每个关键词都能在 Google/GitHub 搜到具体结果"

### 01b_classify_backfill.md（小改）
- 分类判断规则与 01 统一措辞

### 06_keyword_extraction.md（小改）
- 新增宽泛词边界示例："LLM"可以、"AI"太宽泛；"Agent"可以、"技术"太宽泛

### 06b_trend_filtering.md（小改）
- 新增边界判断示例（"发布"移除 vs "产品发布会"保留、"AI"移除 vs "AI Agent"保留）

### 04_action_analysis.md / 04b_action_dedup.md
- 不改动（已是质量最高的 prompt）

### README.md
- 移除个性化 tab 相关条目（03*/05*），更新变量表
