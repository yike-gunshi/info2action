> 一份让用户和 AI 都能按图索骥的配置手册。任何新数据源接入 / token 失效 / 部署到新机器,都从这里开始。
>
> **创建于** 2026-04-19(BF-0419-10 闭环时立项)
> **维护原则** 新增/迁移配置后必须同步更新本文件,否则下次有人又要从 0 摸索

## 0. 凭证存放黄金规则

| 类型 | 放哪 | git 跟踪 | 例子 |
|---|---|---|---|
| **凭证类**(token / key / secret / cookie / 账号密码) | `.env` | ❌ git-ignored | `RESEND_API_KEY`, `LINGOWHALE_AUTH_TOKEN` |
| **业务参数**(关键词 / UP 主清单 / 抓取窗口 / 阈值) | `config/config.json` | ✅ tracked | `lingowhale.max_pages`, `bilibili.up_list` |
| **凭证模板**(占位符,告诉别人需要填什么) | `.env.example` | ✅ tracked | 字段名 + 示例占位 |

**铁律**:
- 凭证 **永远** 不进 git-tracked 文件。如果发现 `config/config.json` 里有 token / key / secret 字段,**这是历史遗留,标 TODO 迁到 .env**。
- 改完 `.env` 必须 **同步改 `.env.example`**(只放占位,不放真值),让别人看得见 schema。
- 改完任何凭证机制,**必须更新本文件对应段落**。

## 1. 首次部署快速上手

```bash
# 1. 复制模板
cp .env.example .env

# 2. 编辑 .env,把 xxxxxxxx 占位换成真实值(取值步骤见下方各数据源段)
vim .env  # 或 code .env

# 3. 跑任何脚本前必须 source(项目惯例,不用 python-dotenv)
set -a && source .env && set +a

# 4. 跑某个 fetch 验证
python3 src/fetch_lingowhale.py
```

⚠️ **每次开新终端都要重新 source .env**,或者把 `set -a; source .env; set +a` 写到你的 `.zshrc`(但只在本项目目录生效)。

## 4. 数据源凭证(按抓取脚本分组)

### 4.1 语鲸订阅 lingowhale ✅(已迁 .env)

**抓取脚本**: `src/fetch_lingowhale.py`
**.env 字段**:

| 字段 | 来源 | 类型 | 失效频率 |
|---|---|---|---|
| `LINGOWHALE_AUTH_TOKEN` | Browser Network Headers | JWT | 30 天 |
| `LINGOWHALE_ACCESS_TOKEN` | 同上 | JWT | 14 天 |
| `LINGOWHALE_UID` | 同上 | UUID-like | 长期不变 |
| `LINGOWHALE_BID` | 同上 | UUID-like(浏览器 fingerprint) | 长期不变 |
| `LINGOWHALE_GUEST_ID` | 同上 | UUID-like | 长期不变 |

**完整取值步骤**:

1. 浏览器(Chrome 推荐)登录 https://lingowhale.com,确保左侧能看到分组列表
2. F12 打开 DevTools → 切到 **Network** 标签
3. Filter 框输入 `subscribe` → 按 Cmd+R / F5 刷新页面
4. Network 列表里点击 `list` 这个请求(URL 含 `/api/feed/v1/user_subscribe/list`)
5. 右侧 Headers 标签 → 滚到 **Request Headers** 段
6. 找以下 5 行,复制值到 `.env` 对应字段:

```
Auth-Token:   eyJhbGciOi...        → LINGOWHALE_AUTH_TOKEN
Access-Token: eyJhbGciOi...        → LINGOWHALE_ACCESS_TOKEN
U-Id:         427303...             → LINGOWHALE_UID
B-Id:         b62a8640...           → LINGOWHALE_BID
Guest-Id:     ef0cf2a6...           → LINGOWHALE_GUEST_ID
```

**验证**:`set -a && source .env && set +a && python3 src/fetch_lingowhale.py | head -5` 看到 `📂 分组: N groups, M channels` 且 N>0 即成功。

**运行时自愈**:

- `_post_json` 遇到语鲸 `code=10010` 时会调用 passport `refresh_token` 端点，并用新 token 重试原请求一次。
- 刷新成功后写入 `data/lingowhale_tokens.json`；后续请求优先读取该文件。文件缺失、损坏或不可读时会容错回退到 `.env`。
- `INFO2ACTION_LINGOWHALE_REGISTRY_ONLY` 默认开启；公众号抓取只使用 sources 注册表中的可抓取频道。仅在显式设为 `false` 时启用 legacy 混合路径。

**失败排查**:
- 首次出现 `code=10010` 后看到 `✅ 已刷新语鲸 token` → 自动自愈成功,无需手工更新 `.env`
- 自动刷新仍失败或重试后继续返回 `code=10010` → 初始 refresh 凭证也已失效,重新走步骤 1-6
- 看到 `语鲸 token store 读取失败,回退 env` → 检查 `data/lingowhale_tokens.json` 内容和读取权限；抓取会继续使用 `.env`
- 看到 `语鲸 token store 写入失败` → 检查 `data/` 写权限；本次保留原错误并等待人工处理
- 看到 `📂 分组: 0 groups, 0 channels` 但无 ❌ → 你账号确实没建分组(或代码问题,看 docs/bugfix/reviews/BF-0419-8.md)
- registry-only 开启时注册表为空或远程读取失败 → 本轮抓取停止并保留错误,不会回退个人订阅或旧 priority 列表

**历史 bug**:
- BF-0419-8(2026-04-19): 代码层缺 `code` 检查,API 错误被静默吞 + 失败覆盖 groups.json 洗快照(已修)
- BF-0419-10(2026-04-19): v1 端点严格校验 `B-Id` / `Guest-Id` / `Origin` / `Referer` headers,代码缺失这些导致 token 不论新旧都返 10010(已修)

### 4.2 Twitter 抓取 ✅(已迁 .env, 2026-04-20)

**抓取脚本**：`src/fetch_x_users.py` 读取 `sources.x_user` 注册表并在每轮遍历全部可抓取账号；`src/fetch_url.py` 只处理用户提交的单条 X 链接。两者底层都调用 `twitter-cli`。
**内容权威**：定时、手动、回填和 QA 抓取入口都只调用注册表账号的 `user-posts`。following、for-you、bookmarks 和关键词搜索不再作为内容入口；旧配置字段与旧磁盘文件不会被 ingest。
**凭证传递**: `.env` → uvicorn env → `_fetch_twitter` 把 env 作为 subprocess env 传给 CLI
**失效频率**: cookie 刷新频率不定,通常账号长期登录保活可用 ~ 周级

**.env 字段**:

| 字段 | 来源 |
|---|---|
| `TWITTER_AUTH_TOKEN` | x.com 登录后浏览器 Cookie 里的 `auth_token` |
| `TWITTER_CT0` | 同上,Cookie 里的 `ct0` |

**取值步骤**:

1. 浏览器登录 https://x.com
2. F12 → Application(或 Storage) → Cookies → `https://x.com`
3. 找 `auth_token` 行,复制值 → `.env` 的 `TWITTER_AUTH_TOKEN`
4. 找 `ct0` 行,复制值 → `.env` 的 `TWITTER_CT0`
5. 重启 uvicorn:`set -a && source .env && set +a && uvicorn src.app:app --port 8080`

**验证**：单链接能力可在前端提交一条 `https://x.com/.../status/...`；订阅抓取需在管理后台添加 X 信源后运行一轮抓取，确认该账号 `last_success_at` 推进。失败常见原因:
- `Twitter 认证已过期` → cookie 过期,回步骤 1 重取
- `推文不存在或已被删除` → tweet 被删或被锁
- submit 无响应 → 检查 uvicorn env 有没有这两个字段:`cat /proc/$(pidof uvicorn)/environ | tr '\0' '\n' | grep TWITTER_`

**历史背景**: BF-迁移前 cookie 硬编码在 `ops/cron_fetch.sh`(git-tracked,历史 commit 已泄露),2026-04-20 迁到 `.env`。旧 token 需手动 rotate(退出 x.com 重登即可作废旧 cookie)。

## 10. 数据运维脚本

### 10.1 本地 → ECS 数据合并同步

**什么时候用**:本地抓了一批新数据,想把它们合并到 ECS 上(不覆盖远程自有数据)。ECS 本身不跑抓取 cron(memory `feedback_no_new_cron_on_ecs`),数据增量全靠手动触发或本地同步。

**脚本位置**:`scripts/ops/sync_local_to_ecs.sh`

**运行**:
```bash
cd /path/to/info2action  # 你的项目根目录
bash scripts/ops/sync_local_to_ecs.sh
```

**会同步的对象**:
- `items` 表 — PK(id) 去重,远程已有的跳过
- `search_keywords` 表 — 复合 PK 去重
- `data/images/` — `rsync --ignore-existing`,远程已有的不动

**不会同步的对象**(保护远程用户数据和本地测试数据):
- 用户账号类:`users` / `sessions` / `invite_codes` / `item_status` / `user_profiles`
- 本地用户衍生数据:`actions` / `action_logs` / `interests` / `interest_matches` / `briefings` / `feedback`
- 本地抓取记录:`fetch_runs` / `health_log` / `asr_usage`
- `.env`:远程已存在则整体跳过,**永远不覆盖**

**安全机制**(每次运行都会经历):
1. 本地先 `PRAGMA wal_checkpoint(TRUNCATE)` 把 WAL 合进主库,避免丢数据
2. Dry-run 预估新增行数 + 按 platform 分布,输出后**等你二次 y 确认**
3. 远程合并前自动备份:`/opt/info-feed/data/feed.db.bak-merge-{YYYYMMDD-HHMM}`
4. `INSERT OR IGNORE` + `busy_timeout=30000` 支持 uvicorn **不停服**并发合并
5. rsync 也先跑 dry-run,数值确认后才真跑
6. 最后验证 `/api/health` 200 + uvicorn PID 不变

**回滚**:如果合并后发现问题,SSH 上去 `mv feed.db.bak-merge-{时间戳} feed.db` 就能回滚。

**典型耗时**(2026-04-20 首次同步):items 新增 8398 条(含 7900 带 AI 摘要)/ 图片新增 1232 个 / 147MB / 耗时 8.2 秒 merge + ~15 秒 rsync。

**对应复盘**:`~/claudecode_workspace/记录/复盘/info2action/2026-04-20-1400-[数据同步]-本地抓取合并到ECS.md`

### 10.2 远程端手动刷新(不新建 cron,靠用户点按钮)

ECS 后端已实现手动刷新接口(`POST /api/fetch` / `POST /api/fetch/quick` / `GET /api/fetch/status`),前端 3 处按钮(FeedSection / TickerBar / HealthPanel)已挂上。**用户登录后点任意刷新按钮即可触发抓取**,无需新开发。

> 与本地同步的区别:本地同步适合"把一批历史数据一次性合并过去";手动刷新适合"让远程自己抓一小段新增"。两者可独立使用,不冲突。

## 11. 给 AI 的提示

如果你是 AI 在帮用户配置 / 排查凭证问题:

1. **永远先读本文件**,不要凭训练记忆猜
2. 用户报"X 抓不到数据" → 来这里查 X 数据源段 → 给完整取值步骤
3. **不要让用户在对话里粘 token 给你**(已记录在 memory: 凭证应直接写 `.env`,不经手 AI)
4. 改 `.env` 后必须告诉用户 `set -a; source .env; set +a` 重新 source
5. 发现 git-tracked 文件里有真值 token → 立刻警告用户 + 加进"待迁移清单"
6. 任何凭证机制变更,**最后一步 = 更新本文件**
7. 用户要把本地数据同步到 ECS → **直接用 `scripts/ops/sync_local_to_ecs.sh`**(§10.1),不要再手写临时命令
